Critical bugs
Bug Singleton client created per-call

_get_client() in ai_caller.py, gap_analyser.py and brand_extractor.py creates a brand-new AsyncOpenAI object on every engine call instead of being truly cached. This defeats connection pooling and leaks TCP connections under load.

Highbackend/ai_caller.py
Bug Model string typo — Gemini

_MODEL_GEMINI defaults to "~google/gemini-pro-latest" (leading tilde). OpenRouter won't recognise this route — all Gemini calls will 404 or return a model-not-found error silently.

Highbackend/ai_caller.py line 36
Bug _result_store eviction is FIFO but unbounded on restarts

The store is an in-memory plain dict. On any server restart (Procfile, Railway, etc.) all stored results are lost — the GET /api/export/pdf/{id} endpoint always returns 404. Users are funnelled to the POST fallback without knowing why.

Mediumbackend/main.py
Bug Fuzzy match false positives in scorer

fuzz.partial_ratio with threshold 85 flags short brand names inside longer words — e.g. "NOW" inside "know" or "Swanson" matched inside "swansong". The conclusion-mention check runs partial_ratio on the last 300 chars of the full normalised text, which often includes unrelated brand fragments, inflating scores.

Mediumbackend/scorer.py
Bug Demo cache engine label mismatch

demo_cache.py uses engine label "Gemini 1.5 Pro" but ai_caller.py emits "Gemini Pro Latest" and constants.js maps only "Gemini Pro Latest". Leaderboard breakdown cards show an unknown engine in the UI when demo data is loaded.

Mediumbackend/demo_cache.pyfrontend/constants.js
Bug run() closure captures stale toast

In App.jsx, run is wrapped in useCallback with an empty dependency array [], but it calls toast.success / toast.error which come from context. If the toast context reference ever changes the closure is stale. Add toast to the deps array.

Mediumfrontend/App.jsx
Bug URL query auto-run fires before React is hydrated

useEffect delays the auto-run by 400ms but depends on the network being ready. On slow connections the fetch to /api/health completes after the auto-run fires, so the SystemStatus spinner overlaps the loading state. Also, if q param contains special chars (e.g. &) it is not sanitised before being passed to run().

Mediumfrontend/App.jsx
Minor PDF background colour blocks WeasyPrint on some systems

body { background: #0a0a0f } in the PDF template requires WeasyPrint's --presentational-hints flag or modern CSS background-color printing support. On older WeasyPrint versions the dark bg is stripped and text becomes invisible (dark text, no bg). Use an explicit print-color-adjust: exact CSS rule.

Features not yet built
User authentication & saved projects

No login system. Results live only in localStorage history (frontend) or the 5-min TTL in-memory cache (backend). Users can't save, revisit, or share their brand tracking across sessions.

Historical trend tracking

Only the latest diagnostic is shown. There's no time-series storage or charting to show how a brand's AI visibility score changes week-over-week — which is the core value-prop for repeat customers.

Competitor monitoring / alerts

No scheduled re-runs or webhook/email alerts when a brand drops out of AI responses or a competitor overtakes it.

Multi-query batch analysis

Users can only submit one query at a time. There's no way to test a brand across 10–20 different shopper queries in one go and see aggregate visibility scores.

Listing text optimisation AI

Gap analysis returns what to fix but doesn't generate the rewritten listing copy. An "AI-rewrite" button that produces an optimised product title + bullet points would close the loop.

More AI engines

Only 3 engines are called (GPT, Claude, Gemini). Perplexity, Mistral, and LLaMA-3 are increasingly used for product discovery and not represented in the score.

CSV / bulk brand import

No way to upload a CSV of competitor brands to track all of them in one report — useful for agencies managing multiple clients.

White-label / branded PDF

PDF is hardcoded with "AEO Diagnostic" branding. No way for agencies to export a co-branded report for their clients.

Getting professional AI responses on a free OpenRouter budget
Use free-tier models as primary engines

OpenRouter provides free daily limits on meta-llama/llama-3.3-70b-instruct:free, mistralai/mistral-7b-instruct:free, and google/gemini-2.0-flash-exp:free. Set these via the MODEL_GPT, MODEL_CLAUDE, MODEL_GEMINI env vars — no code changes needed.

Free
Use Claude via Anthropic's own free tier directly

Anthropic's API gives a free tier for claude-haiku-4-5 calls. Add a direct Anthropic client alongside OpenRouter for gap analysis — Haiku is fast, cheap, and more than capable of JSON gap generation.

Free
Cache aggressively — same query never costs twice

The existing 5-min TTL cache is too short for repeated brand queries. Increase it to 24h for the same query+brand combo. Use a persistent sqlite or diskcache store so results survive restarts — every re-run of a cached query is zero API cost.

Cost saving
Shrink prompts to cut tokens

The system prompt in ai_caller.py is repeated to all 3 engines. Trim it to under 60 tokens. Move heavy instructions into the user turn only when needed. The gap analysis prompt is ~380 tokens — consider a compressed 150-token version for the free path.

Cost saving
Route by task complexity

Use a free/small model (Llama-3 8B) for brand extraction (MODEL_EXTRACT) — it's a pure JSON task with no reasoning needed. Reserve the higher-quality models only for the user-facing gap analysis. This alone can cut your token spend by ~60%.

Cost saving
Fallback to regex-only mode for non-paying users

The regex fallback in brand_extractor.py already exists. Wire a DEMO_MODE=true env flag that skips all LLM calls, uses only regex extraction and the pre-cached demo results. Users can test the UI for free; API calls only fire when they click "Run live analysis".

Free
Code quality improvements
True singleton clients

Move the AsyncOpenAI client into a module-level variable using functools.lru_cache or a _CLIENT = None global pattern so the same client (and its connection pool) is reused across requests.

Persistent result store

Replace the in-memory _result_store with diskcache or sqlite3 so PDF exports survive server restarts. This makes the GET /api/export/pdf/{id} endpoint actually reliable.

Scorer unit tests

The scoring logic is complex and has known false-positive risk. Add pytest tests for edge cases: short brand names (2–3 chars), brands that are substrings of common words, brands not mentioned at all, and brands mentioned only in the conclusion.

Structured streaming progress

The frontend fakes progress with hardcoded setTimeout. Replace with a server-sent events (SSE) endpoint that emits real step completions — "engines done", "brands extracted", "scoring complete". Eliminates the mismatch between fake and real progress.

TypeScript for frontend

The frontend is plain JSX with no type safety. result.leaderboard, result.gap_analysis, and engine breakdown shapes are accessed with no null-guards beyond optional chaining. Migrating to TypeScript would catch these at build time.

Environment validation at startup

Add a Pydantic BaseSettings model that validates all required env vars at import time. Currently a missing OPENROUTER_API_KEY only surfaces as a 503 at request time, which is hard to debug in production.
